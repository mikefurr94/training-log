How the backend works
Serverless functions

Instead of a traditional always-on server, the API runs as serverless functions — small pieces of code that spin up on demand, handle one request, and shut down. Here's what that means concretely.

Traditional server (always-on)
Server boots up
Process starts, port opens, memory allocated
Waiting… (idle, but paying)
CPU and RAM reserved even with 0 requests
Request arrives → handled
Same process handles it
Waiting again…
Still running overnight, on weekends, always
You manage uptime, scaling, crashes
If it dies, the API is down
Serverless (Vercel functions)
Nothing running (zero cost, zero resources)
No process exists between requests
Request arrives → function wakes up
"Cold start" — ~100–300ms to initialize
Handler runs, returns response
Isolated execution, no shared state
Function shuts down (or stays warm briefly)
Vercel may keep it "warm" for ~5 min to avoid cold starts
Vercel manages everything else
Scaling, uptime, retries — not your problem

Deployment model
Is this project containerized?

No — and that's a deliberate non-choice. Here's what the current setup looks like, what it would mean to add Docker, and why the tradeoff favors keeping it simple.

Today — no Docker

What you'd gain
  • Reproducible dev environment — no "works on my machine" Node version issues
  • Platform portability — swap Vercel for any container host
  • Full local stack — run Supabase locally via supabase start
  • Explicit runtime pinning — Node version baked into the image
What you'd lose
  • Vercel's zero-config deploy — push to git and it just works
  • Serverless auto-scaling — containers need sizing and uptime management
  • Simplicity — adds Dockerfile, compose file, CI image build, and registry config
  • Free tier — Vercel hobby is free; container hosting isn't
The verdict
  • For a personal app on Vercel + Supabase — not worth it
  • Node version pinning via .nvmrc solves the main dev concern
  • Containerization makes sense if you ever self-host or need a fully offline dev environment
  • Supabase CLI (supabase start) gives local DB without full Docker setup
Clarification — serverless and containers are independent ideas
Serverless
An execution model. Code spins up per request, scales to zero when idle, you pay per invocation. The platform manages the runtime for you.
orthogonal
Containers
A packaging model. Bundles your code + its runtime into a portable, reproducible image. Says nothing about how or when the code runs.
You can combine them — four real combinations exist:
Not containerized + always-on
Traditional Node/Express server. Install Node, run it, keep it running.
Not containerized + serverless ← this project
Vercel manages the runtime. You just push code. No Docker needed.
Containerized + always-on
Docker container running Express on Railway, Render, or Fly.io. Always running.
Containerized + serverless
Google Cloud Run, AWS Lambda container images. Spins up a container per request, scales to zero.
Why it's built this way
Key architecture decisions

Design choices and the reasoning behind them.

Zustand + localStorage persistence
App works offline and loads instantly. State survives page refreshes without needing a loading spinner. Simple API — no reducers, no boilerplate.
State-based routing over React Router
CalendarPage acts as a hub — switching views is a state change, not a navigation. Preserves scroll position and panel state when switching between modes.
JSONB columns in Supabase
Flexible schema for rapidly evolving features like plans and habits. No migration needed when adding fields. Trades query flexibility for development speed.
Server-side token management
OAuth tokens never touch the frontend. Refresh logic lives in API handlers, preventing token leaks and simplifying the client code.
SSE for AI streaming
Token-by-token streaming without WebSocket complexity. Works with serverless functions (no persistent connections). Native browser EventSource support.
Shared handler code for dev/prod
Same API handler runs as both Express middleware and Vercel serverless function. No code duplication, no environment-specific bugs.
